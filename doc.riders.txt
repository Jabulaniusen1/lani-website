# Lanieats API — Rider Documentation

## Overview

Lanieats is a delivery platform for Nigeria. Riders accept and complete deliveries from merchant stores to customers.

- **Base URL:** `http://localhost:8000/api/v1`
- **Authentication:** Bearer Token (JWT)
- **Content-Type:** `application/json` (use `multipart/form-data` for file uploads)
- **Real-time:** Socket.io

---

## Standard Response Format

Every endpoint returns this structure:

```json
{
  "success": true | false,
  "message": "Human readable message",
  "data": { } // present on success, omitted on error
}
```

Error with validation:
```json
{
  "success": false,
  "message": "Validation failed",
  "errors": [
    { "field": "email", "message": "Please provide a valid email" }
  ]
}
```

---

## Authentication

All protected routes require this header:
```
Authorization: Bearer <token>
```

### Token Strategy

The API uses a two-token model:

| Token | Lifetime | Purpose |
|-------|----------|---------|
| **Access token** (JWT) | `JWT_EXPIRES_IN` (default `7d`) | Sent in `Authorization: Bearer` header on every protected request |
| **Refresh token** (opaque) | `JWT_REFRESH_EXPIRES_IN` (default `30d`) | Stored securely on device; sent to `POST /auth/refresh` to get a new access token |

**Flow:**
1. `POST /auth/login` or `POST /auth/register` → receive `token` (access) + `refreshToken`
2. Use `token` for all API calls
3. When the access token expires, call `POST /auth/refresh` with `{ "refreshToken": "..." }` → receive a new access token **and a new refresh token** (the old refresh token is immediately invalidated — always store the latest one)
4. On logout, call `POST /auth/logout` with the refresh token to invalidate it server-side

## Platform Fee Structure (Rider Earnings)

| Source | Recipient |
|--------|-----------|
| Delivery fee × 90% | Rider |
| Delivery fee × 10% | Platform |

> `RIDER_DELIVERY_FEE_RATE=0.90` — riders keep 90% of the delivery fee. Rider balance = `sum(deliveryFee × 0.90)` across all their delivered + paid orders, minus completed payouts.

## Notifications (Email + Push)

Both email and Expo push notifications are sent for the following events:

| Event | Trigger | Recipients |
|---|---|---|
| Welcome | `POST /auth/register` | Rider |
| Password Reset OTP | `POST /auth/forgot-password` | Rider |
| Rider Approved | Admin approves rider | Rider |
| Rider Rejected | Admin rejects rider | Rider |
| Delivery Code | Order picked up (`PICKED_UP`) | Customer |
| Order Delivered | Delivery status → `DELIVERED` | Customer |

Push notifications are sent via **Expo Push API** (abstracts FCM/APNs). Each notification includes a `data` payload for deep linking (e.g. `{ screen: 'DeliveryDetail', orderId }`).

### Push Notifications — Frontend Setup

**1. Install dependencies**
```bash
npx expo install expo-notifications expo-device
```

**2. Configure `app.json`**
```json
{
  "expo": {
    "plugins": [
      [
        "expo-notifications",
        {
          "icon": "./assets/notification-icon.png",
          "color": "#FF5722"
        }
      ]
    ]
  }
}
```

**3. Register for push and send token to backend** (call this after login)
```js
import * as Notifications from 'expo-notifications';
import * as Device from 'expo-device';

const registerForPush = async (apiClient) => {
  if (!Device.isDevice) return; // must be a real device
  const { status } = await Notifications.requestPermissionsAsync();
  if (status !== 'granted') return;
  const { data: token } = await Notifications.getExpoPushTokenAsync();
  await apiClient.post('/auth/device-token', { token });
};
```

**4. Backend endpoint**

`POST /auth/device-token` *(requires Bearer token)*

```json
{ "token": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]" }
```

The token is stored on the user record and overwritten each login (one active token per user). Riders with no token or denied permission are silently skipped.

---

## CORS

The API restricts cross-origin requests to origins listed in the `ALLOWED_ORIGINS` environment variable (comma-separated). If not set, all origins are allowed (development default).

---

## Rate Limits

Rate limiting is currently disabled.

### Request Timeouts

The server enforces a `SERVER_TIMEOUT_MS` limit (default **30 seconds**) on every request. If a handler does not respond within that window, the server returns:

```json
{ "success": false, "message": "Request timed out" }
```
with HTTP status `503`.

#### GET `/rider/deliveries/available`
Get all deliveries available for pickup. Pass rider coordinates to receive distance and ETA estimates.

**Protected** | `RIDER` only

**Query Params:**
| Param | Type | Description |
|-------|------|-------------|
| `latitude` | float | Rider's current latitude (optional) |
| `longitude` | float | Rider's current longitude (optional) |

**Example:** `GET /rider/deliveries/available?latitude=6.5244&longitude=3.3792`

**Notes:**
- Shows orders with status `PREPARING` or `READY_FOR_PICKUP`
- Shows only unassigned deliveries
- When coordinates are provided, each delivery includes a `distanceInfo` object

**`distanceInfo` object (included when coordinates provided):**
```json
{
  "distanceToRestaurantKm": 1.8,
  "distanceToCustomerKm": 2.3,
  "totalDistanceKm": 4.1,
  "etaToRestaurantMin": 5,
  "etaToCustomerMin": 6,
  "estimatedTotalMin": 11,
  "label": "4.1 km · ~11 min"
}
```

> ETA is calculated using an average speed of 25 km/h. `distanceInfo` is `null` if coordinates are not provided or if the restaurant/delivery address has no saved coordinates.

#### PATCH `/rider/deliveries/:deliveryId/status`
Update delivery status.

**Protected** | `RIDER` only

**Request Body — marking PICKED_UP:**
```json
{
  "status": "PICKED_UP",
  "latitude": 6.5244,
  "longitude": 3.3792
}
```

**Request Body — marking DELIVERED:**
```json
{
  "status": "DELIVERED",
  "confirmationCode": "4823",
  "latitude": 6.5244,
  "longitude": 3.3792
}
```

**Allowed statuses:** `PICKED_UP` → `DELIVERED` | `FAILED`

**Delivery confirmation code flow:**
1. When rider marks status as `PICKED_UP`, a 4-digit code is generated automatically
2. The code is sent to the **customer** via the `delivery_code_issued` socket event
3. The code is also returned to the **rider** in the API response (`data.deliveryCode`)
4. When marking `DELIVERED`, the rider must include `confirmationCode` in the request body
5. If the code doesn't match → `400 "Incorrect confirmation code. Cannot hand over order"`

**Response when PICKED_UP `200`:**
```json
{
  "success": true,
  "message": "Delivery status updated",
  "data": {
    "delivery": { ...delivery object },
    "deliveryCode": "4823"
  }
}
```

**Socket events emitted:**
- `delivery_status_updated` → to customer
- `delivery_code_issued` → to customer order room (when PICKED_UP, includes the 4-digit code)
- `rider_location_updated` → to customer (if lat/lng provided)
- `order_delivered` → to customer's personal room (when DELIVERED)

> When a delivery is marked `DELIVERED`, the rider's `isAvailable` is only reset to `true` if they have no other active deliveries remaining.

#### POST `/rider/deliveries/:deliveryId/issue`
Report an issue with a delivery.

**Protected** | `RIDER` only — must be the assigned rider

**Request Body:**
```json
{
  "type": "CANT_FIND_CUSTOMER",
  "description": "Been waiting 10 minutes at the address, no answer"
}
```

**Issue types:**
| Type | Description |
|------|-------------|
| `CANT_FIND_CUSTOMER` | Rider at address but customer unreachable |
| `WRONG_ADDRESS` | Address is incorrect or doesn't exist |
| `CUSTOMER_REFUSED` | Customer refused to accept the order |
| `ITEM_DAMAGED` | Item was damaged before or during transit |
| `RESTAURANT_NOT_READY` | Rider arrived but order is not ready |
| `OTHER` | Any other issue — use `description` field |

**Notes:**
- `description` is optional but recommended for `OTHER` type
- Socket event `delivery_issue_reported` is emitted to the admin room

**Response `201`:**
```json
{
  "success": true,
  "message": "Issue reported successfully",
  "data": {
    "issue": {
      "id": "uuid",
      "deliveryId": "uuid",
      "riderId": "uuid",
      "type": "CANT_FIND_CUSTOMER",
      "description": "Been waiting 10 minutes at the address, no answer",
      "resolvedAt": null,
      "createdAt": "2026-03-11T15:00:00.000Z"
    }
  }
}
```

### Delivery Status Flow

```
UNASSIGNED  → Created with order, waiting for a rider
ASSIGNED    → Rider accepted, heading to store
PICKED_UP   → Rider collected the order
DELIVERED   → Delivered to customer
FAILED      → Delivery failed
```

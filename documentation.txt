# Lanieats API — Merchant Documentation

## Overview

Lanieats is a delivery platform for Nigeria covering food, pharmacy, and supermarket orders. Merchants manage stores (restaurants, pharmacies, supermarkets), their catalogs, and fulfill orders.

- **Base URL:** `http://localhost:8000/api/v1`
- **Authentication:** Bearer Token (JWT)
- **Content-Type:** `application/json` (use `multipart/form-data` for file uploads)
- **Payment Gateway:** Paystack (NGN)
- **Real-time:** Socket.io

---

## Standard Response Format

Every endpoint returns this structure:

```json
{
  "success": true | false,
  "message": "Human readable message",
  "data": { } // present on success, omitted on error
}
```

Error with validation:
```json
{
  "success": false,
  "message": "Validation failed",
  "errors": [
    { "field": "email", "message": "Please provide a valid email" }
  ]
}
```

---

## Authentication

All protected routes require this header:
```
Authorization: Bearer <token>
```

### Token Strategy

The API uses a two-token model:

| Token | Lifetime | Purpose |
|-------|----------|---------|
| **Access token** (JWT) | `JWT_EXPIRES_IN` (default `7d`) | Sent in `Authorization: Bearer` header on every protected request |
| **Refresh token** (opaque) | `JWT_REFRESH_EXPIRES_IN` (default `30d`) | Stored securely on device; sent to `POST /auth/refresh` to get a new access token |

**Flow:**
1. `POST /auth/login` or `POST /auth/register` → receive `token` (access) + `refreshToken`
2. Use `token` for all API calls
3. When the access token expires, call `POST /auth/refresh` with `{ "refreshToken": "..." }` → receive a new access token **and a new refresh token** (the old refresh token is immediately invalidated — always store the latest one)
4. On logout, call `POST /auth/logout` with the refresh token to invalidate it server-side

---

## Platform Fee Structure

| Source | Recipient |
|--------|-----------|
| 10% of order subtotal | Platform |
| 90% of order subtotal | Merchant |
| Service charge (₦200 flat per order) | Platform (full amount) |
| Delivery fee | Rider |

> `MERCHANT_COMMISSION_RATE=0.90` — the merchant's available payout balance is calculated as `sum(subtotal × 0.90)` across all their delivered + paid orders, minus any completed payouts.

> `MIN_PAYOUT_AMOUNT=1000` — minimum payout is read from this env variable server-side and exposed via `GET /payouts/config`.

## Notifications (Email + Push)

Both email and Expo push notifications are sent for the following events:

| Event | Trigger | Recipients |
|---|---|---|
| Welcome | `POST /auth/register` | Merchant |
| Password Reset OTP | `POST /auth/forgot-password` | Merchant |
| New Order Received | Paystack `charge.success` webhook | Merchant |
| Order Cancelled (by customer) | `DELETE /orders/:id/cancel` | Merchant |
| Merchant Approved | Admin approves merchant | Merchant |
| Merchant Rejected | Admin rejects merchant | Merchant |
| Payout Submitted / Processing | `POST /payouts/request` | Merchant |

Push notifications are sent via **Expo Push API** (abstracts FCM/APNs). Each notification includes a `data` payload for deep linking (e.g. `{ screen: 'OrderDetail', orderId }`).

### Push Notifications — Frontend Setup

**1. Install dependencies**
```bash
npx expo install expo-notifications expo-device
```

**2. Configure `app.json`**
```json
{
  "expo": {
    "plugins": [
      [
        "expo-notifications",
        {
          "icon": "./assets/notification-icon.png",
          "color": "#FF5722"
        }
      ]
    ]
  }
}
```

**3. Register for push and send token to backend** (call this after login)
```js
import * as Notifications from 'expo-notifications';
import * as Device from 'expo-device';

const registerForPush = async (apiClient) => {
  if (!Device.isDevice) return; // must be a real device
  const { status } = await Notifications.requestPermissionsAsync();
  if (status !== 'granted') return;
  const { data: token } = await Notifications.getExpoPushTokenAsync();
  await apiClient.post('/auth/device-token', { token });
};
```

**4. Backend endpoint**

`POST /auth/device-token` *(requires Bearer token)*

```json
{ "token": "ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]" }
```

The token is stored on the user record and overwritten each login (one active token per user). Merchants with no token or denied permission are silently skipped.

---

## CORS

The API restricts cross-origin requests to origins listed in the `ALLOWED_ORIGINS` environment variable (comma-separated). If not set, all origins are allowed (development default).

---

## Rate Limits

Rate limiting is currently disabled.

### Request Timeouts

The server enforces a `SERVER_TIMEOUT_MS` limit (default **30 seconds**) on every request. If a handler does not respond within that window, the server returns:

```json
{ "success": false, "message": "Request timed out" }
```
with HTTP status `503`.

### 🔐 AUTH

#### POST `/auth/register`
Register a new merchant account.

**Public** | No token required

**Content-Type:** `multipart/form-data` (required when uploading a CAC document; `application/json` is fine otherwise)

**Request Body (form fields):**
```
firstName             string   required
lastName              string   required
email                 string   required
phone                 string   required  (Nigerian format: 080XXXXXXXX or +234XXXXXXXXX)
password              string   required
role                  string   "MERCHANT"
merchantType          string   "RESTAURANT" | "PHARMACY" | "SUPERMARKET"  (default: RESTAURANT)
isBusinessRegistered  string   "true" | "false"  (whether the business is CAC-registered)
cacDocument           file     Required when isBusinessRegistered=true (image or PDF, max 10MB)
```

**Response `201`:**
```json
{
  "success": true,
  "message": "Registration successful",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "a1b2c3d4...",
    "user": {
      "id": "uuid",
      "firstName": "Jabulani",
      "lastName": "Doe",
      "email": "jab@test.com",
      "phone": "08012345678",
      "role": "MERCHANT",
      "isVerified": false,
      "isActive": true,
      "createdAt": "2026-02-25T00:00:00.000Z"
    }
  }
}
```

**Notes:**
- A `Merchant` profile is auto-created with the given `merchantType` (defaults to `RESTAURANT`)
- If `isBusinessRegistered=true`, the `cacDocument` file field is **required** — omitting it returns `400`
- CAC document is uploaded to Cloudinary under the `cac-documents` folder; the URL is stored on the `Merchant` record as `cacDocumentUrl`
- If `isBusinessRegistered=false`, no document is required and `cacDocumentUrl` will be `null`
- The merchant account must be approved by an admin before it can operate (see `isApproved`)

#### PATCH `/orders/:id/status`
Update order status.

**Protected** | `MERCHANT` only

**Request Body — confirming/preparing:**
```json
{
  "status": "PREPARING",
  "estimatedPrepTime": 25
}
```

**Request Body — cancellation:**
```json
{
  "status": "CANCELLED",
  "cancelReason": "Item out of stock"
}
```

**Allowed statuses for merchant:** `PREPARING` | `READY_FOR_PICKUP` | `CANCELLED`

| Field | Required | Description |
|-------|----------|-------------|
| `status` | Yes | New order status (`PREPARING`, `READY_FOR_PICKUP`, or `CANCELLED`) |
| `estimatedPrepTime` | No | Minutes to prepare (1–180) — set when moving to `PREPARING` |
| `cancelReason` | **Yes if cancelling** | Reason shown to the customer |

> `CONFIRMED` is set automatically when payment is received via webhook — it cannot be set manually. The merchant's first manual action is always `PREPARING`.

**Socket events emitted:**
- `order_status_updated` → to customer tracking the order
- `delivery_available` → to all riders (when status is `PREPARING` or `READY_FOR_PICKUP`)
- `order_cancelled` → to customer's personal room (when merchant cancels, includes reason)

---

### Order Status Flow

```
PENDING           → Order placed, awaiting payment
CONFIRMED         → Payment received (automatic — set by webhook)
PREPARING         → Merchant is preparing the order (riders get notified)
READY_FOR_PICKUP  → Order is ready, rider should be arriving
OUT_FOR_DELIVERY  → Rider picked up the order
DELIVERED         → Order complete
CANCELLED         → Cancelled by customer or merchant
```

#### GET `/analytics/:restaurantId/overview`
Get key stats for a restaurant over a time period.

**Protected** | `MERCHANT` only

**Query Params:**
| Param | Default | Options |
|-------|---------|---------|
| `period` | `7d` | `1d`, `7d`, `30d` |

**Response `200`:**
```json
{
  "success": true,
  "message": "Stats fetched",
  "data": {
    "period": "7d",
    "totalOrders": 42,
    "deliveredOrders": 38,
    "cancelledOrders": 2,
    "completionRate": 90,
    "revenue": 95000,
    "avgOrderValue": 2500,
    "previousPeriod": {
      "totalOrders": 35,
      "deliveredOrders": 30,
      "completionRate": 85,
      "revenue": 78000,
      "avgOrderValue": 2228
    },
    "changes": {
      "totalOrders": 20,
      "deliveredOrders": 27,
      "completionRate": 6,
      "revenue": 22,
      "avgOrderValue": 12
    }
  }
}
```

> `changes` values are percentage differences vs the previous equivalent period. `null` means no data in the previous period.

## 🔌 Socket.io — Merchant Events

### Connection

```js
import { io } from 'socket.io-client';
const socket = io('http://localhost:8000');
```

### Rooms to Join

```js
socket.emit('join_merchant_room', restaurantId); // join after login, once per store
```

### Events the SERVER sends to MERCHANT

| Event | Payload | Description |
|-------|---------|-------------|
| `new_order` | `{ orderId, total, itemCount, preview, createdAt }` | New order placed (before payment) |
| `order_confirmed` | `{ orderId, message }` | Payment received, order auto-confirmed |
| `order_cancelled` | `{ orderId, reason, message }` | Customer cancelled the order |

## 🗂️ Data Models

### Merchant
```
id, userId, businessName, businessPhone,
merchantType (RESTAURANT | PHARMACY | SUPERMARKET),
isBusinessRegistered, cacDocumentUrl, isApproved, createdAt, updatedAt
```

> `isBusinessRegistered` — set at signup. If `true`, the merchant submitted a CAC document.
> `cacDocumentUrl` — Cloudinary URL of the CAC registration document (`null` if not registered).

### Restaurant (Store)
```
id, merchantId, name, description, address, city, state, latitude, longitude,
logoUrl, coverUrl, isOpen, isApproved, openingTime, closingTime, rating, totalReviews,
isBusy, busyMessage, avgPrepTime, restaurantType, createdAt, updatedAt
```

### MenuItem / Product
```
id, restaurantId, categoryId, name, description, price, imageUrl, isAvailable,
requiresPrescription, stockQuantity
```

### Order
```
id, userId, restaurantId, addressId, status, paymentStatus, paymentRef,
subtotal, deliveryFee, serviceCharge, total, note,
estimatedPrepTime, deliveryCode, cancelReason, cancelledBy, createdAt, updatedAt
```

### MerchantBankAccount
```
id, merchantId, accountName, accountNumber, bankName, bankCode,
recipientCode, isVerified, createdAt, updatedAt
```

### PayoutRequest
```
id, merchantId, amount, status (PENDING | PROCESSING | COMPLETED | FAILED),
reference, narration, failureReason, requestedAt, processedAt
```

## Default Store Categories (Auto-seeded on store creation)

**RESTAURANT:** Appetizers & Starters, Rice Dishes, Swallows, Soups & Stews, Grills & BBQ, Chicken, Seafood, Burgers, Pizza, Pasta, Shawarma & Wraps, Snacks, Sides, Salads, Desserts, Drinks & Beverages

**PHARMACY:** Prescription Medicines, Over-the-Counter Medicines, Pain Relief, Cold, Cough & Flu, Malaria Care, Vitamins & Supplements, First Aid & Wound Care, Digestive Health, Skin Care Treatments, Women's Health, Baby Care, Medical Devices

**SUPERMARKET:** Fresh Produce, Meat & Poultry, Seafood, Dairy & Eggs, Bread & Bakery, Pantry Staples, Rice, Pasta & Noodles, Snacks & Confectionery, Beverages, Frozen Foods, Household Supplies, Personal Care

Run `npm run seed` to backfill existing stores. The seed is idempotent — safe to run multiple times.

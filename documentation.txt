# Lanieats API Documentation

## Overview

Lanieats is a delivery platform for Nigeria covering food, pharmacy, and supermarket orders. It has three client applications — Customer App, Merchant App, and Rider App — all powered by a single backend API.

- **Base URL:** `http://localhost:8000/api/v1`
- **Authentication:** Bearer Token (JWT)
- **Content-Type:** `application/json`
- **Payment Gateway:** Paystack (NGN)
- **Real-time:** Socket.io
- **Email:** Nodemailer via Gmail SMTP

---

## Email System

All transactional emails are sent via Gmail SMTP using `nodemailer`. Emails are sent non-blocking (`.catch` logged to console) so they never break API responses.

### Configuration

Add to `.env`:
```
SMTP_USER=your-gmail@gmail.com
SMTP_PASS=your-16-digit-app-password
```

> Gmail requires an **App Password** (not your login password).
> Enable 2FA on your Google account, then generate one at:
> https://myaccount.google.com/apppasswords

### Files
- `src/config/email.js` — nodemailer transporter (Gmail)
- `src/services/email.service.js` — all email templates and send functions

### Emails Sent

| Event | Recipient | Trigger |
|---|---|---|
| Welcome | Customer / Merchant / Rider | `POST /auth/register` |
| Password Reset OTP | Any user | `POST /auth/forgot-password` |
| Order Confirmed | Customer | Paystack `charge.success` webhook |
| New Order Received | Merchant | Paystack `charge.success` webhook |
| Order Being Prepared | Customer | Merchant updates status → `PREPARING` |
| Order Ready for Pickup | Customer | Merchant updates status → `READY_FOR_PICKUP` |
| Order Cancelled (by merchant) | Customer | Merchant updates status → `CANCELLED` |
| Order Cancelled (by customer) | Merchant | `DELETE /orders/:id/cancel` |
| Delivery Code | Customer | Rider marks delivery → `PICKED_UP` |
| Order Delivered | Customer | Rider marks delivery → `DELIVERED` |
| Merchant Approved | Merchant | Admin approves merchant |
| Merchant Rejected | Merchant | Admin rejects merchant |
| Rider Approved | Rider | Admin approves rider |
| Rider Rejected | Rider | Admin rejects rider |
| Payout Submitted / Processed | Merchant | `POST /payout/request` |
| Abandoned Cart Reminder | Customer | Automated cron job (every 15 min) |

### Abandoned Cart Job

An order is considered **abandoned** when:
- `status = PENDING` and `paymentStatus = PENDING` (order placed, payment never completed)
- Created more than **30 minutes** ago
- `abandonedReminderSentAt` is null (reminder not yet sent)

The job runs every 15 minutes via `node-cron`, sends one reminder email per order, then stamps `abandonedReminderSentAt` so the customer is never emailed twice for the same order.

**Files:**
- `src/jobs/abandonedCart.job.js` — cron job logic
- `prisma/schema.prisma` — `Order.abandonedReminderSentAt DateTime?` field
- `prisma/migrations/…add_abandoned_reminder_to_order/` — migration

### Database Seeding

Run to backfill default categories for stores that already exist:

```bash
npm run seed
```

This seeds the following default categories:
For `RESTAURANT` stores:
`Appetizers & Starters`, `Rice Dishes`, `Swallows`, `Soups & Stews`, `Grills & BBQ`, `Chicken`, `Seafood`, `Burgers`, `Pizza`, `Pasta`, `Shawarma & Wraps`, `Snacks`, `Sides`, `Salads`, `Desserts`, `Drinks & Beverages`

For `PHARMACY` stores:
`Prescription Medicines`, `Over-the-Counter Medicines`, `Pain Relief`, `Cold, Cough & Flu`, `Malaria Care`, `Vitamins & Supplements`, `First Aid & Wound Care`, `Digestive Health`, `Skin Care Treatments`, `Women's Health`, `Baby Care`, `Medical Devices`

For `SUPERMARKET` stores:
`Fresh Produce`, `Meat & Poultry`, `Seafood`, `Dairy & Eggs`, `Bread & Bakery`, `Pantry Staples`, `Rice, Pasta & Noodles`, `Snacks & Confectionery`, `Beverages`, `Frozen Foods`, `Household Supplies`, `Personal Care`

The seed is idempotent — safe to run multiple times, it skips categories that already exist.

## Standard Response Format

Every endpoint returns this structure:

```json
{
  "success": true | false,
  "message": "Human readable message",
  "data": { } // present on success, omitted on error
}
```

Error with validation:
```json
{
  "success": false,
  "message": "Validation failed",
  "errors": [
    { "field": "email", "message": "Please provide a valid email" }
  ]
}
```

## Authentication

All protected routes require this header:
```
Authorization: Bearer <token>
```

### Token Strategy

The API uses a two-token model:

| Token | Lifetime | Purpose |
|-------|----------|---------|
| **Access token** (JWT) | `JWT_EXPIRES_IN` (default `7d`) | Sent in `Authorization: Bearer` header on every protected request |
| **Refresh token** (opaque) | `JWT_REFRESH_EXPIRES_IN` (default `30d`) | Stored securely on device; sent to `POST /auth/refresh` to get a new access token |

**Flow:**
1. `POST /auth/login` or `POST /auth/register` → receive `token` (access) + `refreshToken`
2. Use `token` for all API calls
3. When the access token expires, call `POST /auth/refresh` with `{ "refreshToken": "..." }` → receive a new access token **and a new refresh token** (the old refresh token is immediately invalidated — always store the latest one)
4. On logout, call `POST /auth/logout` with the refresh token to invalidate it server-side

### Roles
| Role | Description |
|------|-------------|
| `CUSTOMER` | Places orders, tracks delivery |
| `MERCHANT` | Manages stores (restaurants, pharmacies, supermarkets) and their catalogs, fulfills orders |
| `RIDER` | Accepts and completes deliveries |
| `ADMIN` | Platform management |

## Rate Limits

Rate limiting is currently disabled.

### Request Timeouts

The server enforces a `SERVER_TIMEOUT_MS` limit (default **30 seconds**) on every request. If a handler does not respond within that window, the server returns:

```json
{ "success": false, "message": "Request timed out" }
```
with HTTP status `503`.

## Platform Fee Structure

| Source | Recipient |
|--------|-----------|
| 10% of order subtotal | Platform |
| 90% of order subtotal | Merchant |
| Service charge (₦200 flat per order) | Platform (full amount) |
| Delivery fee (configurable rate) | Rider |

> `MERCHANT_COMMISSION_RATE=0.90` is set in `.env`. The merchant's available payout balance is calculated as `sum(subtotal × 0.90)` across all their delivered + paid orders, minus any completed payouts.

> `MIN_PAYOUT_AMOUNT=1000` is set in `.env`. The minimum payout is read from this variable server-side and exposed via `GET /payouts/config` so the mobile app never needs a hard-coded value.

> `RIDER_DELIVERY_FEE_RATE=0.90` is set in `.env`. Riders keep 90% of the delivery fee — the platform takes a 10% commission. Rider balance = `sum(deliveryFee × RIDER_DELIVERY_FEE_RATE)` across all their delivered + paid orders, minus completed payouts.

### 🔐 AUTH

#### POST `/auth/register`
Register a new user.

**Public** | No token required

**Content-Type:** `multipart/form-data` (required when uploading a CAC document; can use `application/json` for non-merchant or unregistered-business merchants)

**Request Body (form fields):**
```
firstName       string   required
lastName        string   required
email           string   required
phone           string   required  (Nigerian format: 080XXXXXXXX or +234XXXXXXXXX)
password        string   required
role            string   "CUSTOMER" | "MERCHANT" | "RIDER"  (default: CUSTOMER)
merchantType    string   "RESTAURANT" | "PHARMACY" | "SUPERMARKET"  (MERCHANT only, default: RESTAURANT)
isBusinessRegistered  boolean  "true" | "false"  (MERCHANT only — whether the business is CAC-registered)
cacDocument     file     Required when isBusinessRegistered=true (image or PDF, max 10MB)
```

**Response `201`:**
```json
{
  "success": true,
  "message": "Registration successful",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "refreshToken": "a1b2c3d4...",
    "user": {
      "id": "uuid",
      "firstName": "Jabulani",
      "lastName": "Doe",
      "email": "jab@test.com",
      "phone": "08012345678",
      "role": "CUSTOMER",
      "isVerified": false,
      "isActive": true,
      "createdAt": "2026-02-25T00:00:00.000Z"
    }
  }
}
```

**Notes:**
- On `MERCHANT` registration, a `Merchant` profile is auto-created with the given `merchantType` (defaults to `RESTAURANT`)
- On `RIDER` registration, a `Rider` profile is auto-created
- Phone must be a valid Nigerian number (e.g. 080XXXXXXXX or +234XXXXXXXXX)
- `merchantType` and `isBusinessRegistered` are ignored for non-MERCHANT roles
- If `isBusinessRegistered=true`, the `cacDocument` file field is **required** — omitting it returns `400`
- CAC document is uploaded to Cloudinary under the `cac-documents` folder; the URL is stored on the `Merchant` record as `cacDocumentUrl`
- If `isBusinessRegistered=false`, no document is required and `cacDocumentUrl` will be `null`

#### POST `/auth/refresh`
Exchange a refresh token for a new access token + rotated refresh token.

**Public** | No access token needed

**Request Body:**
```json
{
  "refreshToken": "<the refresh token received at login>"
}
```

**Response `200`:**
```json
{
  "success": true,
  "message": "Token refreshed",
  "data": {
    "token": "eyJhbGci...",
    "refreshToken": "<new refresh token — store and replace the old one>"
  }
}
```

> Returns `401` if the refresh token is missing, invalid, or expired. Refresh tokens are rotated on every use — the old one is invalidated immediately.

---

#### POST `/auth/logout`
Invalidate a refresh token (server-side). Call this when the user logs out.

**Public** | No access token needed

**Request Body:**
```json
{
  "refreshToken": "<the refresh token to invalidate>"
}
```

**Response `200`:**
```json
{
  "success": true,
  "message": "Logged out successfully"
}
```

> Silently succeeds even if the token is already expired or not found.

#### POST `/auth/forgot-password`
Request a password reset code. A 6-digit OTP is generated and should be sent to the user via email or SMS.

**Public** | No token required

**Request Body:**
```json
{
  "email": "jab@test.com"
}
```

**Response `200`:**
```json
{
  "success": true,
  "message": "If that email is registered, a reset code has been sent"
}
```

> Always returns `200` regardless of whether the email exists — this prevents email enumeration. The reset code expires in **15 minutes**. In development, the code is logged to the server console.

**Response `200`:**
```json

  "success": true,

```

> Returns `400` if the code is wrong or expired. Code is invalidated after use.

### 🍖 MENU / PRODUCTS

All catalog routes are nested under the store: `/restaurants/:restaurantId/menu`

This single set of routes serves all merchant types. The fields accepted and returned differ by `merchantType`:

| Field | Restaurant | Pharmacy | Supermarket |
|-------|-----------|----------|-------------|
| `name`, `price`, `description` | ✅ required | ✅ required | ✅ required |
| `categoryId` | ✅ required | optional | optional |
| `requiresPrescription` | ❌ | ✅ | ❌ |
| `stockQuantity` | ❌ | ✅ required | ✅ required |

> For pharmacy and supermarket, `stockQuantity` is **required** when creating a product. Stock is automatically decremented when orders are placed, and the product is auto-marked unavailable when it reaches 0.

### Order Status Flow

```
PENDING           → Order placed, awaiting payment
CONFIRMED         → Payment received (automatic — set by webhook/verify)
PREPARING         → Merchant is preparing/packing the order (riders get notified)
READY_FOR_PICKUP  → Order is ready, rider should be arriving
OUT_FOR_DELIVERY  → Rider picked up the order
DELIVERED         → Order complete
CANCELLED         → Cancelled by customer or merchant
```

#### PATCH `/rider/deliveries/:deliveryId/status`
Update delivery status.

**Request Body — marking PICKED_UP:**
```json
{
  "status": "PICKED_UP",
  "latitude": 6.5244,
  "longitude": 3.3792
}
```

**Request Body — marking DELIVERED:**
```json
{
  "status": "DELIVERED",
  "confirmationCode": "4823",
  "latitude": 6.5244,
  "longitude": 3.3792
}
```

**Allowed statuses:** `PICKED_UP` → `DELIVERED` | `FAILED`

**Delivery confirmation code flow:**
1. When rider marks status as `PICKED_UP`, a 4-digit code is generated automatically
2. The code is sent to the **customer** via the `delivery_code_issued` socket event
3. The code is also returned to the **rider** in the API response (`data.deliveryCode`)
4. When marking `DELIVERED`, the rider must include `confirmationCode` in the request body
5. If the code doesn't match → `400 "Incorrect confirmation code. Cannot hand over order"`

**Response when PICKED_UP `200`:**
```json
{
  "success": true,
  "message": "Delivery status updated",
  "data": {
    "delivery": { ...delivery object },
    "deliveryCode": "4823"
  }
}
```

**Socket events emitted:**
- `delivery_status_updated` → to customer
- `delivery_code_issued` → to customer order room (when PICKED_UP, includes the 4-digit code)
- `rider_location_updated` → to customer (if lat/lng provided)
- `order_delivered` → to customer's personal room (when DELIVERED)

> When a delivery is marked `DELIVERED`, the rider's `isAvailable` is only reset to `true` if they have no other active deliveries remaining.

#### POST `/payments/webhook`
Paystack webhook endpoint.

**Public** | Called by Paystack servers only

**Notes:**
- Validates Paystack signature via HMAC SHA-512
- On `charge.success`: sets `paymentStatus` → `PAID` and `status` → `CONFIRMED`
- Emits `order_confirmed` to the merchant's store room
- Emits `order_status_updated` to the customer tracking the order
- Always returns `200` to Paystack
- Set this URL in your Paystack dashboard → Settings → Webhooks

**Local testing with ngrok:**
1. Start the API:
```bash
npm run dev
```
2. Start ngrok in another terminal:
```bash
npm run ngrok
```
3. Copy the HTTPS forwarding URL from ngrok (example: `https://abcd-1234.ngrok-free.app`).
4. In Paystack Dashboard (Test mode), set webhook URL to:
```text
https://abcd-1234.ngrok-free.app/api/v1/payments/webhook
```
5. Restart/update webhook URL whenever ngrok gives you a new tunnel URL.

## 🔌 Socket.io Real-Time Events

### Connection

```js
import { io } from 'socket.io-client';

const socket = io('http://localhost:8000');
```

### Joining Rooms

Each app must join the appropriate room after login:

```js
// Customer app
socket.emit('join_customer_room', userId);
socket.emit('track_order', orderId); // when viewing an order

// Merchant app
socket.emit('join_merchant_room', restaurantId);

// Rider app
socket.emit('join_rider_room', riderId);

// Customer browsing a restaurant page (to receive busy mode updates)
socket.emit('join_restaurant_customers', restaurantId);
```

---

### Events Reference

#### Events the SERVER emits to CLIENT

| Event | Room | Payload | Description |
|-------|------|---------|-------------|
| `new_order` | `restaurant_{id}` | `{ orderId, total, itemCount, preview, createdAt }` | New order placed |
| `order_confirmed` | `restaurant_{id}` | `{ orderId, message }` | Payment received, order auto-confirmed |
| `order_status_updated` | `order_{id}` | `{ orderId, status, estimatedPrepTime, cancelReason, updatedAt }` | Order status changed |
| `order_cancelled` | `customer_{id}` or `restaurant_{id}` | `{ orderId, reason, message }` | Order was cancelled |
| `rider_assigned` | `order_{id}` | `{ orderId, message }` | Rider accepted the delivery |
| `delivery_available` | broadcast | `{ orderId, status, restaurantName }` | Order ready for rider pickup |
| `delivery_status_updated` | `order_{id}` | `{ orderId, deliveryStatus, updatedAt }` | Rider updated delivery status |
| `delivery_code_issued` | `order_{id}` | `{ orderId, deliveryCode, message }` | 4-digit code issued when rider picks up the order |
| `rider_location_updated` | `order_{id}` | `{ orderId, latitude, longitude }` | Rider location changed |
| `order_delivered` | `customer_{id}` | `{ orderId, message }` | Order successfully delivered |
| `restaurant_busy_updated` | `restaurant_customers_{id}` | `{ restaurantId, isBusy, busyMessage }` | Restaurant busy mode toggled |
| `delivery_issue_reported` | `admin` | `{ deliveryId, riderId, type, description, createdAt }` | Rider reported a delivery issue |

#### Events the CLIENT emits to SERVER

| Event | Payload | Description |
|-------|---------|-------------|
| `join_customer_room` | `userId` | Customer joins personal room |
| `join_merchant_room` | `restaurantId` | Merchant joins restaurant room |
| `join_rider_room` | `riderId` | Rider joins personal room |
| `track_order` | `orderId` | Join specific order room for live updates |
| `join_restaurant_customers` | `restaurantId` | Join restaurant browsing room (busy mode updates) |

## 🗂️ Data Models Reference

### User
```
id, email, phone, firstName, lastName, role, isVerified, isActive, createdAt, updatedAt
```

### Merchant
```
id, userId, businessName, businessPhone, merchantType (RESTAURANT | PHARMACY | SUPERMARKET),
isBusinessRegistered, cacDocumentUrl, isApproved, createdAt, updatedAt
```

> `isBusinessRegistered` — set at signup. If `true`, the merchant submitted a CAC document.
> `cacDocumentUrl` — Cloudinary URL of the uploaded CAC registration document (`null` if not registered).

### Restaurant
```
id, merchantId, name, description, address, city, state, latitude, longitude,
logoUrl, coverUrl, isOpen, isApproved, openingTime, closingTime, rating, totalReviews,
isBusy, busyMessage, avgPrepTime, createdAt, updatedAt
```

### MenuItem
```
id, restaurantId, categoryId, name, description, price, imageUrl, isAvailable,
requiresPrescription, stockQuantity
```

### Order
```
id, userId, restaurantId, addressId, status, paymentStatus, paymentRef,
subtotal, deliveryFee, serviceCharge, total, note,
estimatedPrepTime, deliveryCode, cancelReason, cancelledBy,
createdAt, updatedAt
```

### Delivery
```
id, orderId, riderId, status, pickedUpAt, deliveredAt, photoProofUrl, createdAt, updatedAt
```

### Rider
```
id, userId, vehicleType, vehiclePlate,
nin, photographUrl, dateOfBirth, gender, homeAddress, isOnboarded,
isAvailable, isApproved,
currentLatitude, currentLongitude, rating, totalReviews, totalDeclined
```

### MerchantBankAccount
```
id, merchantId, accountName, accountNumber, bankName, bankCode,
recipientCode, isVerified, createdAt, updatedAt
```

### PayoutRequest
```
id, merchantId, amount, status (PENDING | PROCESSING | COMPLETED | FAILED),
reference, narration, failureReason, requestedAt, processedAt
```

### RiderBankAccount
```
id, riderId, accountName, accountNumber, bankName, bankCode,
recipientCode, isVerified, createdAt, updatedAt
```

### RiderPayoutRequest
```
id, riderId, amount, status (PENDING | PROCESSING | COMPLETED | FAILED),
reference, narration, failureReason, requestedAt, processedAt
```

### DeliveryIssue
```
id, deliveryId, riderId, type (CANT_FIND_CUSTOMER | WRONG_ADDRESS | CUSTOMER_REFUSED |
ITEM_DAMAGED | RESTAURANT_NOT_READY | OTHER), description, resolvedAt, createdAt
```
